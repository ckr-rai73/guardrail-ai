# ADR-098-01: Repository Manifest Specification

## Context
Guardrail.ai extends its trust boundary to the software supply chain to neutralize repository-borne configuration attacks, such as malicious `.claude-plugin/marketplace.json` or `.cursor/settings.json` which can hijack developer environments.

## Decision
We introduce a signed manifest file, `guardrail-manifest.json`, which must reside in the root of any repository analyzed by Guardrail.ai or interacting with the Guardrail.ai agent mesh.

### Schema: guardrail-manifest.json
- **`version`**: Manifest schema version.
- **`repository_id`**: Unique identifier for the repository.
- **`allowed_tools`**: Whitelist of allowed AI tools and their specific configuration overrides.
    - Example: `{ "mcp_servers": ["cloudflare-mcp", "postgres-mcp"], "auto_approve_flags": false }`
- **`signatures`**: Array of ed25519 cryptographic signatures by authorized repository administrators or Guardrail.ai Vanguard Nodes.

### Signature Enforcement (Crypto Edge)
The backend `ManifestAuditor` module will parse all discovered local configs (e.g., hidden AI tool folders) and strictly enforce that they align with the configurations permitted by `guardrail-manifest.json`. The `guardrail-manifest.json` itself must have a valid ed25519 signature in its `signatures` block. If the signature is invalid or missing, the tool configs are treated as malicious.

## Consequences
- Repositories without a signed manifest, or with tool configs exceeding the allowed limits, will be assigned a `RepositoryTrustScore` < 0.8.
- These repositories will be quarantined within the Edge Sandbox.
- Violations will trigger telemetry alerts to the `GlobalAuditor` and `VectorClockLedger`.
